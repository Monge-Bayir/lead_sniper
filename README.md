# lead_sniper
```
Отчёт по заданию

Подход

Решение построено как воспроизводимый data-pipeline:
поиск → сбор → фильтрация → доказательство → нормализация → экспорт.

Основной принцип — автоматизация и прозрачные правила, а не ручной отбор компаний.

⸻

Как нашли “сиды” (первичный список компаний)

В качестве стартовой точки использовались вакансии hh.ru:
	•	вакансии отражают реальный масштаб поддержки;
	•	HH API позволяет стабильно собирать данные;
	•	вакансии легко агрегируются на уровне работодателя.

Собирались вакансии по запросам:
	•	служба поддержки
	•	call-центр
	•	contact center
	•	customer support
	•	оператор поддержки

Из вакансий извлекались vacancy_id и employer_id, после чего работодатели дедуплицировались.

⸻

Как определяли признаки поддержки

На уровне вакансий

Использовались regex-паттерны по названию и описанию вакансий.

Позитивные признаки:
	•	поддержк*
	•	call-центр / контакт-центр
	•	helpdesk / support
	•	техподдержка
	•	чат / тикеты / SLA
	•	L1 / L2

Негативные признаки (исключения):
	•	продавец
	•	кассир
	•	менеджер по продажам
	•	директор магазина
	•	кредитный аналитик
	•	врач

⸻

На уровне сайтов компаний

Фиксировались булевы признаки:
	•	has_support_email
	•	has_contact_form
	•	has_online_chat
	•	has_messengers
	•	has_support_section
	•	has_kb_or_faq
	•	mentions_24_7

⸻

Как доказывали “10+ сотрудников поддержки”

Уровень A (сильное доказательство)

Прямое числовое подтверждение:
	•	«в поддержке 25 специалистов»
	•	«контакт-центр 200 операторов»

В этом случае:
	•	support_team_size_min = N
	•	сохранялся фрагмент текста и ссылка

⸻

Уровень B (консервативная оценка)

Если прямого числа нет, компания считалась ≥10 при наличии признаков:
	•	поддержка 24/7 (сменный график);
	•	несколько уровней поддержки (L1/L2);
	•	несколько вакансий поддержки одновременно;
	•	упоминание “крупного контакт-центра”.

В этом случае:
	•	support_team_size_min = 10
	•	в support_evidence описывалась логика

⸻

Как мёржили компании
	•	основной идентификатор: employer_id (HH)
	•	сайт компании: извлекался со страницы работодателя HH
	•	ИНН:
	•	с сайта компании (если доступно);
	•	через Rusprofile (по домену и названию);
	•	DaData как дополнительный источник

⸻

Что получилось / что не получилось

Что удалось
	•	Собрано 163 компании с доказанным размером поддержки ≥10
	•	Реализован полностью автоматизированный и воспроизводимый пайплайн
	•	Прозрачная логика evidence (A/B)
	•	Чистый CSV без дублей и мусора

Ограничения
	•	Не у всех компаний удалось автоматически получить ИНН (бренд ≠ юрлицо)
	•	Не использовался headless-браузер для JS-рендеринга сайтов
	•	Холдинги с несколькими юрлицами не агрегировались

⸻

Роль LLM

LLM не использовались в финальной версии.

Причина:
	•	задачи фильтрации и evidence надёжно решаются правилами;
	•	повышается воспроизводимость и объяснимость.

LLM могли бы использоваться для:
	•	классификации страниц сайта;
	•	извлечения evidence из длинных текстов.
```
